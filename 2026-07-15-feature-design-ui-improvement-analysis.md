# 십 묶음 택배 터미널 기능·디자인·UI 개선 분석

**작성일:** 2026-07-15
**대상:** 현재 공개 버전 v1.2.0의 소스 코드, 학습 문서, 테스트, 기존 디자인 개선 계획
**목적:** 이미 구현된 장점을 유지하면서 초등학교 1~2학년 학생이 더 쉽게 이해하고, 교실의 작은 화면과 보조기술 환경에서도 안정적으로 사용할 수 있도록 다음 개선 후보를 정리합니다.

> 이 문서는 분석과 후속 개선 계획만 다룹니다. 이번 작업에서는 앱 코드, 배포본, 업데이트 내역을 변경하지 않습니다.

## 1. 종합 결론

현재 앱은 핵심 학습 목표, 고정 9개 미션, 점수 없는 활동, 개인정보 비저장, 묶기·풀기 수량 불변식이 잘 연결된 완성도 높은 MVP입니다. 특히 3/9 미션에서 `지금 할 일`과 조작 순서를 직접 보여 주도록 개선되어, 이전의 가장 큰 진행 방해 요소는 해결되어 있습니다. 밝은 파스텔 색상과 택배 삽화도 저학년 대상에 적합합니다.

지금 단계에서 전면 재설계는 필요하지 않습니다. 다음 버전에서는 아래 다섯 가지를 먼저 다루는 것이 효과가 큽니다.

1. 이미 로직에 있으나 화면에는 없는 `현재 미션 다시 시작` 기능을 제공합니다.
2. 묶기·풀기·되돌리기 버튼의 색과 형태를 서로 다르게 해 조작 의미를 즉시 구분합니다.
3. 모바일 헤더와 작업대를 더 짧게 만들어 세로 스크롤 부담을 줄입니다.
4. 진행 표시를 단순한 `3 / 9`에서 `완료·현재·남은 미션`이 보이는 형태로 개선합니다.
5. 시작 화면뿐 아니라 작업대·문제·완료·대화상자까지 자동 접근성 및 반응형 검사를 넓히고, VoiceOver·물리 터치·200% 확대를 수동 검수합니다.

치명적인 기능 오류나 즉시 수정이 필요한 P0 문제는 확인되지 않았습니다. 아래 P1은 장애 수정이 아니라, 다음 개선 릴리스에서 우선 적용할 항목을 뜻합니다.

## 2. 현재 구현 상태와 강점

### 학습 구조

- 0, 9, 10, 19, 20, 42, 40, 99, 100을 고정 순서로 학습합니다.
- `전체 수 = 십 묶음 수 × 10 + 낱개 수`가 모든 미션에서 유지됩니다.
- 7개와 10개를 비교하는 짧은 연습 뒤 본 미션으로 들어가므로 첫 진입 부담이 낮습니다.
- 묶기, 풀기, 되돌리기, 숫자 쓰기, 수 읽기, 자리 관계를 한 흐름 안에서 경험합니다.
- 점수, 제한 시간, 순위, 무작위 문제를 사용하지 않아 저학년 학습자가 실수를 탐색으로 받아들일 수 있습니다.
- 100은 `십 묶음 10개 = 100개 = 백 1개`까지만 관찰하고 백 묶음 조작은 범위에서 제외해 교육과정 경계를 지킵니다.

### 기능과 오류 회복

- 조작할 수 없는 이유를 `낱개 10개가 있어야 해요`처럼 구체적으로 설명합니다.
- 3/9 및 7/9 미션은 필요한 마지막 조작 순서를 올바르게 수행하면 통과하도록 판정합니다.
- 오답을 입력해도 기존 입력을 지우지 않아 다시 생각할 기회를 줍니다.
- 전체 초기화에는 확인 대화상자가 있으며, 업데이트 내역을 열어도 학습 상태가 유지됩니다.
- 학생 정보, 브라우저 저장소, 서버, 외부 API, 분석 도구를 사용하지 않습니다.

### 디자인과 UI

- 밝은 크림·하늘·민트·살구색을 사용하며 시스템 다크 모드와 무관하게 밝은 화면을 유지합니다.
- 시작 화면의 생성 삽화는 학습 맥락을 전달하는 장식으로 사용하고, 정확한 `낱개 10개 = 십 묶음 1개` 표현은 코드 기반 모형으로 다시 제공합니다.
- 십 묶음과 낱개를 서로 다른 영역으로 나누고, 글자와 수량을 함께 표시합니다.
- 주요 버튼의 높이가 최소 48px이고 키보드 초점 표시가 뚜렷합니다.
- 작은 화면에서 1열로 전환되고 `prefers-reduced-motion`, 강제 색상 모드 대응 코드가 있습니다.

### 테스트와 유지보수

- 이번 분석 중 단위·컴포넌트 테스트 **76개가 모두 통과**했습니다.
- 브라우저 테스트 **24개도 모두 통과**했으며, 전체 9개 미션, 키보드 입력, 업데이트 대화상자, 첫 화면 자동 접근성, 320·375·768·1280px 가로 넘침 검사가 포함되어 있습니다.
- TypeScript·TSX·CSS 파일이 모두 500줄 미만으로 분리되어 있습니다.
- 화면에 업데이트 내역 버튼과 날짜별 변경 기록이 유지되고 있습니다.

## 4. 기능 개선 상세

### 4.1 현재 미션 다시 시작

가장 먼저 적용할 기능입니다. 내부 상태와 리듀서에는 이미 현재 미션을 초기 상태로 되돌리는 동작이 구현되어 있으므로 새 학습 로직을 만들 필요가 없습니다.

권장 동작은 다음과 같습니다.

- 위치: `한 단계 되돌리기` 옆 또는 작업대 하단의 보조 버튼
- 이름: `이 미션 다시 시작`
- 범위: 현재 미션의 묶음 상태, 조작 기록, 답안, 피드백만 초기화
- 유지: 전체 완료 기록과 이전 미션 진행
- 확인창: 조작이나 입력이 있을 때만 표시
- 안내 문구: `이번 미션의 택배만 처음 모습으로 돌아가요.`

이 기능은 실수한 학생이 전체 9개 미션을 다시 시작하는 상황을 막아 줍니다.

### 4.2 진행률과 미션 지도

현재 `3 / 9`는 간결하지만 저학년에게는 “3번 문제”인지 “3개를 완료했다”는 뜻인지 모호할 수 있습니다. 다음과 같이 명시적인 상태를 권장합니다.

- 화면 표시: `2개 완료 · 지금 3번째 · 6개 남음`
- 시각 표시: 9개의 작은 택배 칸 중 완료는 체크, 현재는 굵은 테두리, 남은 칸은 빈 테두리
- 접근 가능한 이름: `전체 9개 미션 중 2개 완료, 현재 3번째 미션`
- 모바일: 9칸 대신 `2 완료 · 3번째 · 6 남음` 한 줄 요약

진행 지도는 점수나 보상이 아니라 위치 안내이므로 현재의 점수 없는 운영 원칙과 충돌하지 않습니다.

### 4.3 완료 후 복습과 선택 연습

본 미션 진행 중 자유 이동을 허용하면 학습 순서가 깨질 수 있습니다. 따라서 복습은 모든 미션을 끝낸 정리표에서 여는 편이 안전합니다.

- 완료한 미션 제목과 핵심 개념을 목록으로 표시합니다.
- `10`, `40`, `100` 같은 경계 미션을 다시 열 수 있게 합니다.
- 복습 완료 뒤에는 정리표로 돌아옵니다.
- 복습 여부와 결과는 저장하지 않습니다.
- 보너스 연습은 정답 생성형보다 검수된 고정 문제를 사용합니다.

### 4.4 도움말의 위치

현재 `이번에 할 순서` 카드는 내용이 좋습니다. 다만 카드와 실제 버튼 사이에 수직 간격과 다른 정보가 끼면 연결이 약해질 수 있습니다.

권장 구조는 다음과 같습니다.

1. 지금 할 일
2. 바로 해당하는 조작 버튼
3. 바뀐 택배 모습과 전체 수
4. 순서 완료 후 `묶음 확인`

즉, 안내 문구를 더 늘리기보다 안내와 조작의 물리적 거리를 줄이는 것이 중요합니다.

## 5. 디자인 및 UI 개선 상세

### 5.1 유지할 시각 체계

다음 요소는 앱의 정체성과 학습 이해에 도움이 되므로 유지합니다.

- 크림색 바탕과 하늘색·민트·살구색의 밝은 팔레트
- 시작 화면의 택배 터미널 삽화
- 십 묶음=하늘색 계열, 낱개=살구색 계열의 구분
- 굵은 숫자와 큰 터치 영역
- 코드로 정확하게 그린 십 묶음·낱개 모형
- 색상과 문자를 함께 사용하는 상태 안내

### 5.2 줄일 시각 요소

저학년용 디자인에서 “귀여움”은 도움이 되지만 모든 요소가 강조되면 중요한 행동을 찾기 어려워집니다.

- 작업대에서는 별 장식과 배경 도형을 최소화합니다.
- 카드마다 다른 그림자를 주지 않고 `기본 카드`, `현재 행동`, `완료` 세 단계만 사용합니다.
- 이모지는 브랜드와 보조 설명에만 두고 핵심 조작에는 일관된 상자 아이콘을 사용합니다.
- 알약형 배지는 짧은 상태값에만 사용하고 긴 설명은 일반 문장으로 둡니다.
- 모든 카드의 모서리를 똑같이 크게 둥글게 하기보다, 학습 모형은 20~24px, 작은 안내는 12~16px처럼 계층을 구분합니다.

### 5.3 버튼 의미 체계

권장 색상 역할은 다음과 같습니다.

| 기능 | 권장 표현 | 이유 |
|---|---|---|
| 다음 단계·확인 | 진한 파란색 채움 | 앱 전체의 주 행동으로 일관성 유지 |
| 묶기 | 파란색 또는 하늘색 채움 + 묶인 상자 아이콘 | 십 묶음 영역과 연결 |
| 풀기 | 살구색 채움 + 열린 상자 아이콘 | 낱개 영역과 연결 |
| 되돌리기 | 흰색 바탕 + 회색 테두리 + 되돌림 아이콘 | 주 행동과 경쟁하지 않음 |
| 미션 다시 시작 | 흰색 바탕 + 주황 테두리 | 되돌리기보다 영향이 큰 동작임을 알림 |
| 전체 처음부터 | 대화상자 안의 경고색 | 화면 상시 강조를 피함 |

색만으로 구분하지 않고 버튼 이름과 아이콘을 반드시 함께 둡니다.

### 5.4 모바일·확대 화면

작은 화면과 200% 확대에서는 가로 폭보다 세로 길이와 정보 순서가 더 큰 문제입니다.

- 헤더를 2줄 이내로 제한합니다.
- 작업대의 관계 문장은 `4묶음 × 10 + 2낱개 = 42개`를 먼저 보여 주고 긴 설명은 그 아래 둡니다.
- 숫자 키패드는 터치 기기에서 유지하되 물리 키보드 사용 시 접을 수 있게 할 수 있습니다.
- 수량 모형의 최대 높이를 제한하고, 많은 묶음은 개별 상자와 `× 10` 표기를 함께 사용해 과도하게 길어지지 않게 합니다.
- 현재 행동과 확인 버튼이 같은 화면 구간에 보이도록 구성합니다.

## 6. 접근성 및 실제 기기 검수

### 6.1 자동 검사를 넓힐 화면

현재 첫 화면 중심 검사를 다음 상태까지 확대합니다.

- 연습 7개 화면
- 연습 10개 묶기 화면
- 3/9 묶기 작업대
- 7/9 풀기→묶기 작업대
- 숫자 입력 화면과 화면 키패드
- 읽기·자리 관계 선택 화면
- 미션 완료 화면
- 배송 정리표
- 업데이트 내역과 초기화 확인 대화상자

각 상태에서 다음을 확인합니다.

- serious·critical axe 위반 0개
- 접근 가능한 이름 중복 또는 누락 없음
- 초점 이동 순서가 화면의 읽기 순서와 일치
- 상태 안내가 한 번만 자연스럽게 낭독
- 320px과 375px에서 가로 넘침 없음
- 키보드만으로 모든 조작과 대화상자 닫기 가능

### 6.2 수동 확인 항목

자동화하기 어려워 후속 확인으로 남겨야 하는 항목은 다음과 같습니다.

#### macOS VoiceOver 실제 낭독

- Safari에서 VoiceOver를 켜고 첫 화면부터 정리표까지 실제로 이동합니다.
- 화면이 바뀔 때 제목이 먼저 낭독되는지 확인합니다.
- `지금 할 일`, 조작 성공, 오류, 순서 완료가 중복 없이 한 번씩 읽히는지 확인합니다.
- 십 묶음·낱개·전체 수가 시각적 상자 개수에 의존하지 않고 문장으로 전달되는지 확인합니다.
- 대화상자를 열면 초점이 안으로 이동하고 닫으면 원래 버튼으로 돌아오는지 확인합니다.

#### 물리 터치

- iPad 또는 실제 터치 화면에서 탭, 연속 탭, 스크롤을 확인합니다.
- 묶기와 풀기를 빠르게 연속 입력해도 중복 조작이나 레이아웃 흔들림이 없는지 확인합니다.
- 숫자 키패드, 지우기, 선택 카드의 실제 터치 범위가 충분한지 확인합니다.
- 스크롤 중 버튼이 실수로 눌리지 않는지 확인합니다.

#### 브라우저 200% 확대

- Safari와 Chrome에서 200% 확대 후 시작, 작업대, 문제, 완료, 대화상자를 확인합니다.
- 텍스트가 카드 밖으로 잘리지 않는지 확인합니다.
- 헤더 버튼과 진행 표시가 겹치지 않는지 확인합니다.
- 대화상자 내부가 스크롤 가능하고 닫기 버튼에 접근할 수 있는지 확인합니다.
- 고정·sticky 요소가 본문이나 확인 버튼을 가리지 않는지 확인합니다.

## 7. 테스트 보강 제안

현재 테스트는 학습 로직을 충분히 보호하지만, 다음 회귀 테스트를 추가하면 UI 변경의 안전성이 높아집니다.

1. `이 미션 다시 시작`이 현재 미션만 초기화하고 이전 완료 기록은 유지하는지 확인합니다.
2. 3/9와 7/9에서 다음 행동 버튼이 시각 클래스와 접근 가능한 문구로 함께 강조되는지 확인합니다.
3. 각 대표 화면에서 axe serious·critical 위반이 없는지 확인합니다.
4. 시작 화면뿐 아니라 작업대, 키패드, 정리표에서 320px 가로 넘침이 없는지 확인합니다.
5. 200% 확대와 유사한 CSS viewport에서 헤더와 확인 버튼이 겹치지 않는지 확인합니다.
6. 진행 표시가 완료 수, 현재 위치, 남은 수를 정확하게 계산하는지 확인합니다.
7. 시작 화면에서는 불필요한 전체 초기화 확인창을 열지 않는지 확인합니다.
8. 앱 표시 버전과 변경 이력의 최신 버전이 일치하는지 확인합니다.

## 11. 최종 판단

추천 우선순위는 `오류 회복 → 조작 의미 → 모바일 정보 구조 → 접근성 검수 → 복습 기능` 순서입니다.
